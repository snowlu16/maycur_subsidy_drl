## 一、 需求调研与基础逻辑黄金确认准则 (Checklist)

在承接或编写新的补贴 DRL 规则前，**务必向项目顾问/客户确认以下 4 大基本要素**，避免因规则不清晰而反复重构脚本：

### 1. 扣减场景与对应维度确认
- **是否扣减**：需明确是否存在扣减场景（例如常驻地扣减、有招待费打车费等关联单据扣减、连续行程天数扣减等）。
- **扣减挂载维度**：
  - **单据级 / 行程级**：如“行程上勾选了某选项则扣减”、“超出 X 天后单天扣减”。
  - **费用类型级**：当报销/申请单上存在某种费用（如招待费、餐饮费），需提供该关联费用的**具体业务编码**。
  - **扣减频次与模式**：需明确是“有一笔关联费用扣减一次，还是多笔多次扣减 / 有则只扣一次减半 / 归零”。
- **自定义选项字段匹配**：若在行程上增加了下拉选项（如【是否派车】、【出行标准选项】），**必须提供**：
  - 行程上该选项字段的业务编码（例如 `CF1`）；
  - 该字段在“自定义档案”里的具体子项编码（例如 `是=001`，`否=002`）。

### 2. 补贴标准表行、列配置规范（死规矩）
- **列维度必配城市**：在每刻后台配置补贴标准表时，**不管企业是否有不同城市的金额标准分层（哪怕全公司全国都是统一标准 `100元/天`），在列维度上也必须配置并勾选一个【城市维度】（如“全部城市”或“其他城市”）**！若列上无城市维度，底层 `getAllowanceStandardInfo` 将无法正常匹配返回值。
- **行维度常用类型**：
  - 城市分组（一线/二线/其他）；
  - 业务场景维度（配置在行程上）；
  - 时间维度（工作日 / 节假日 / 周末）；
  - 自定义档案维度（如员工职级、是否派车等）。

### 3. 汇总方式表述规范（请严谨使用以下三种标准术语之一）
1. **按天生成（一天一条）**：费用明细日期为单天（`YYYY-MM-DD`）。如果有多行程合并或报销人与同行参与人，需明确是“所有人合并算一条”还是“报销人与参与人各自算一天一条”。
2. **按行程汇总（一个行程一段一条）**：费用明细日期为该段行程的起止日期区间（`start ~ end`）。如有参与人，明确各一条还是合并一条。
3. **全单汇总为一条（暂不支持开箱即用）**：单据内所有行程合为一条总计区间（若需此场景，请先和底层产品或集成确认是否开启特殊定制）。

### 4. 必提供参数清单
- **补贴标准编码 (`bizCode`)**：系统“补贴标准”配置页面对应的文本业务编码（如 `ALLOWANCE_TRAFFIC_STD`）。
- **生成明细费用类型编码 (`feeCode`)**：算完生成的 `AllowanceResult` 应归属的报销费用类型编码（如 `ALLOWANCE_TRAFFIC_FEE`）。

## 二、 特殊业务场景与系统配置最佳实践

### 1. 【核心打破误区】自定义档案作为补贴标准行维度的极简写法
#### 场景需求
当企业需要根据每个行程上的自定义选择（如“是否派车”、“职级类别”）匹配不同金额标准时（如：派车补贴0元，不派车补贴80元）。
#### 底层真理与开发指南
- **DRL 脚本中完全不需要写任何 `customFormValues` 判断逻辑！**
- **底层原理**：由费控补贴引擎在执行 `allowanceService.getAllowanceStandardInfo(...)` 时，底层自动检查补贴标准表中的行维度“自定义档案”在当前行程 `TravelRoute` 中是否存在并吻合，吻合则返回该标准金额，不吻合返回 null。
#### 正确配置步骤
1. 后台新建自定义档案 —— 例如【是否派车】（含选项 `是`、`否`）；
2. 在“设置 -> 维度配置”内新增业务场景维度表，绑定【是否派车】自定义档案；
3. 补贴标准表行维度勾选并选择【自定义档案-是否派车】，分别录入金额；
4. 申请单/报销单行程表单上新增下拉组件，引用该【是否派车】自定义档案即可！

### 2. 多行程目的地的默认标准计算规则（20260716 最新规范）
若员工单条行程填写了多个目的地城市（如 `杭州 -> 上海 -> 苏州`）：
- **默认机制**：底层默认按照**最后一个目的地城市**的标准匹配金额；
- **可灵活配置**：在补贴标准与计算规则配置页面，支持勾选切换为按照：**所有目的地内的最高标准、最低标准、第一个目的地标准、最后一个目的地标准**。

### 3. 外币补贴币种与申请单币种冲突处理 (关键费控限制)
#### 冲突现象
若申请单申请币种为 `CNY`，但匹配的海外差旅补贴标准内配置为外币（如 `EUR` / `USD`），当单据提交后，系统会因表单“申请币种”组件的约束，强行将补贴费用的币种显示或更新为 `CNY`（数字不变但币种错变），导致金额直接错乱！
#### 解决方案（唯一官方推荐方法）
- **申请单表单上直接删除“申请币种”字段组件**！
- 改造后机制：无论补贴标准是外币还是人民币，底层会将标准金额以**该外币币种**生成 `AllowanceResult` 费用明细；插入单据费用表后，费用控制系统会自动根据当前系统汇率将该外币金额**转换为公司抬头的本币金额**（无需在 DRL 中手动乘汇率）。

### 4. 节约补贴 (差旅住宿节约差额奖励) 实现方案
#### 场景与要求
当员工实际住宿费用低于标准住宿限额时，将省下金额的一定比例（例如差额的 `50%`）作为“节约补贴”发放给员工。
#### 配置与实施步骤
1. **后台准备**：在“补贴标准”中，建一套和【多维费控-住宿标准】维度一模一样的标准表（例如叫 `BIZ_HOTEL_STD`）；
2. **DRL 逻辑编写**：
   - 获取该段行程对应城市日期的住宿标准上限金额；
   - 遍历关联单据中的实际住宿费用明细；
   - 校验：若实际住宿费用 `< 住宿标准金额 * 80%`（或规定门槛），则执行公式：
     ```java
     BigDecimal savedAmount = standardHotelAmount.subtract(actualHotelAmount);
     BigDecimal allowanceAmount = savedAmount.multiply(new BigDecimal("0.50")).setScale(2, BigDecimal.ROUND_HALF_UP);
     ```

## 三、 专属集成 API 与进阶开发提示

### 1. 动态获取外币/目的地补贴标准设置的币种
在复杂国际出差或外币分流场景中，如需动态拿到当前匹配的标准对应的是哪个币种字符串（如 `EUR` / `CNY`），调用：
```java
String currency = allowanceService.getDestinationAllowanceStandardCcy($startDate, $allowanceStandardInfo);
```
拿到 `currency` 后，通过 `allowanceResult.setCurrency(currency)` 显式传入返回对象中。

### 2. 申请单明细挂载行程显示隐式规则
当“多维费控 -> 申请单 -> 费用明细”的组件设置配置为 **【需要在行程下填写申请明细】** 时：
- **约束**：算出来的补贴如果未关联行程 Code，单据保存页面**绝不会显示任何明细**！
- **正确代码规范**：
  ```java
  // 务必向结果中注入该条费用归属的行程 code
  allowanceResult.setTravelRouteCode($travelRoute.getCode());
  ```
  *(注：若不想强挂行程，需后台将申请单组件设置为“申请明细单独填写”。)*

### 3. 海外跨天/时差计算偏差（时区处理）
若员工提单所在地在海外（非东八区 `UTC+8`），或者行程起止跨越国际时区，可能会造成底层通过毫秒时间戳算自然天数（`Days.daysBetween`）时多出 1 天或少 1 天。遇到特殊时区单据计算偏差，需先联系集成研发确认底层租户是否开启了跨时区标准化归一处理。

## 四、 高频 BUG 与排错与阿里云 SLS 底层日志定位 (Troubleshooting)

### 1. 前端常见报错与排查自查清单
| 异常表现 | 排查方向与原因 | 快速解决方法 |
| :--- | :--- | :--- |
| **算出来金额全是 0 / 单条费用被跳过** | `1)` 脚本里配置了 `if(amount > 0) insert` 导致金额为0被过滤；<br>`2)` 补贴标准表列维度**没选【城市】**；<br>`3)` 编码 (`bizCode`/`feeCode`) 在产线被改动。 | 确认后台是否选了全部城市列；检查 DRL 日志中 `standardAmount` 打印值。 |
| **提示补贴“生成成功”，但页面明细区域一片空白** | 申请单设置了“需要在行程下填写申请明细”，但 DRL 没写 `setTravelRouteCode(...)`。 | 在 `then` 中为 `allowanceResult` 补充赋值 `setTravelRouteCode($travelRoute.getCode())`。 |
| **外币补贴无法生成 (人民币却正常)** | `1)` 系统未开通/配置多币种与汇率模块；<br>`2)` 补贴标准内的城市没有包含该海外目的城市。 | 检查多币种开通状态及外币城市标准覆盖范围。 |
| **点击测试/计算补贴，没有任何反应/提示空/根本没进入日志** | 报销单属于无行程纯自定义补贴单据，且**既没有行程 (`TravelRoute`为空)，也没有任何现有费用明细 (`Expense`为空)**。底层引擎默认此单据无差旅/无实体消费，直接判定不进入执行规则。 | 在报销单明细列表中先手工添加/关联**至少一笔任意费用明细**（确保 `费用明细不为空`），随后再次点击测试/计算即可成功触发。 |
| **控制台/SLS报错：Unable to build expression for consequence: offset -1, count 1, length 0 (或运行时异常)** | `1)` `then ... end` 内部 `//` 单行注释截断了 AST 语法树或紧邻 `end`；<br>`2)` `new AllowanceResult(...)` 使用了官方 4 个重载之外的组合；<br>`3)` 误用了 `consumeDate.withTimeAtStartOfDay()`，部分集群的旧版 `joda-time` 不支持该方法导致抛异常报错。 | `1)` 将注释换行，保证 `end` 独立；<br>`2)` 严格对照官方传参；<br>`3)` **直接去掉 `withTimeAtStartOfDay()`**，直接传入 `$submittedAt` 或 `DateTime.now()` 即可执行成功。 |

### 2. 阿里云 SLS 底层 Java 异常堆栈排查步骤 (极重中之重)
#### 核心背景
当上传或运行 DRL 脚本时，每刻控制台只会做简单的语法检验（Syntax Check）。如果脚本由于**运行时异常（Runtime Exception）**——如对 `null` 对象调用方法、类型强制转换 `ClassCastException`、或者 `JodaTime` 格式解析异常而计算崩溃，**前端页面只会提示“计算/生成补贴失败”或直接返回空**，不会暴露底层 Java 堆栈。
此时，开发及运维人员**必须**通过阿里云 SLS（日志服务）定位真正代码死穴。

#### SLS 集群与环境对应字典
- **正常生产主集群 (`ng`)**：查 Logstore -> `maycur-web_log`；（若在灰度环境则查 `grayscale-cluster`）
- **预发与 UAT 环境 (`ng-uat`)**：查 Logstore -> `ng-uat-log`；（灰度查 `cluster2-nguat`）
- **钉钉专有云生产版**：查 Logstore -> `dt-k8s-prod-service`

#### 标准排查日志 4 步法
1. **确认接口与租户**：确认触发计算的请求 URI（通常为 `allowance/generate` 或 `reimburse/save`）以及企业租户唯一标识 `entCode`；
2. **提取 TraceId**：在网络面板 (Network) Header 或微服务调用网关日志中，获取当前计算请求对应的全局追踪 ID `traceId`；
3. **定位 SLS Logstore**：进入阿里云 SLS 控制台，打开对应环境的 Logstore；
4. **组合查询语法获取异常堆栈**：在搜索栏中精准过滤规则引擎服务日志：
   ```text
   <你的traceId> and __topic__ : rule-service
   ```
   *（示例：`da92c2ffb50a3966 and __topic__ : rule-service`）*
   展开查询结果的 `exception` 或 `message` 字段，直接查看具体的代码行数及底层抛出的 Java Exception！
